# 딩파고 AI 입찰공고 분석 시스템 v1.0 — 평가 파이프라인 가이드

작성일: 2026-04-25

---

## 1. 개요

`eval/` 파이프라인은 AI 추출 결과를 `data/ground_truth/`에 저장된 정답과 대조하여
필드별 정확도, Source Grounding, Hallucination Rate를 측정합니다.

---

## 2. 평가 실행 방법

### 기본 실행 (VCR replay 모드)

```bash
PYTHONIOENCODING=utf-8 python -m eval.run_eval \
  --dataset data/ground_truth \
  --out reports/
```

### 전체 옵션

```
--dataset  ground_truth 루트 경로 (기본: data/ground_truth)
--out      리포트 출력 경로 (기본: reports/)
--api      v1.0 API base URL (기본: http://localhost:8000, replay 모드에서는 미사용)
--limit    평가 건수 제한 (기본: 제한 없음)
--vcr      VCR 모드: replay | record | disabled (기본: replay)
```

### 실제 API 호출 (record 모드)

```bash
# 서버 먼저 기동
GEMINI_API_KEY=your_key python -m uvicorn app.main:app --host 0.0.0.0 --port 8000

# 평가 실행 (실제 Gemini 호출 + cassette 저장)
PYTHONIOENCODING=utf-8 python -m eval.run_eval \
  --dataset data/ground_truth \
  --out reports/ \
  --vcr record
```

---

## 3. VCR 모드 설명 (비용 가드)

| 모드 | 동작 | 비용 |
|---|---|---|
| `replay` | cassette 파일 재생. Gemini 미호출. | 무료 |
| `record` | 실제 API 호출 후 cassette 저장. | Gemini 과금 발생 |
| `disabled` | 실제 API 호출. cassette 미저장. | Gemini 과금 발생 |

**기본값은 `replay`**로 CI/CD에서 추가 비용 없이 회귀 테스트를 실행합니다.

cassette 파일 위치: `tests/fixtures/gemini_cassettes/{notice_id}.json`

새 샘플을 추가하면 `--vcr record`로 한 번 cassette를 생성한 뒤 커밋하세요.

## 4. 평가 메트릭 정의

### 4.1 가중평균 정확도 (weighted_avg)

- required=true 필드 가중치 **2.0**, optional 필드 가중치 **1.0**
- 필드별 점수: 정확 1.0, 부분 0.5~1.0, 불일치 0.0
- 양쪽 null이면 both_null (1.0으로 처리)

### 4.2 Source Grounding (source_grounding_avg)

- 추출 값에 대한 근거 원문이 실제 공고 텍스트에 포함되어 있는지 비율
- 높을수록 AI가 원문에 기반해 추출 (낮으면 hallucination 위험)

### 4.3 Hallucination Rate

- source 없이 값이 채워진 필드 비율
- 0에 가까울수록 좋음

### 4.4 Top10 Failures

- 필드별 정확도 하위 10개
- 프롬프트 보강 우선순위 결정에 사용

## 5. 회귀 테스트 임계값 정책

| 시점 | 임계값 | 비고 |
|---|---|---|
| D7 (1주 완료) | 70% | 첫 임계값. CI 차단 기준. |
| 2주차 | 85% | 프롬프트 보강 후 목표 |
| 1개월 | 95% | 프로덕션 배포 기준 |

임계값은 `REGRESSION_THRESHOLD` 환경변수로 오버라이드 가능:

```bash
REGRESSION_THRESHOLD=0.85 python -m pytest tests/test_regression.py -v
```

## 6. 새 필드 추가 시 평가 흐름

```
1. data/columns.csv  → 필드 추가 (key, description, type, required, validator)
2. data/few_shot/    → {key}_examples.md 작성 (T2: positive 2 + negative 1, T3: 2예시)
3. data/ground_truth/{notice_id}/expected.json → 정답 갱신
4. --vcr record      → 새 cassette 생성
5. pytest            → 회귀 테스트 통과 확인
6. git commit        → cassette + few_shot + expected.json 함께 커밋
```

## 7. 리포트 산출물

`reports/{timestamp}/` 아래 3종이 생성됩니다:

| 파일 | 내용 |
|---|---|
| `summary.md` | 전체 점수 + Top10 실패 필드 요약 |
| `per_field.csv` | 필드별 정확도·correct·mismatch 수치 |
| `failures/` | mismatch·partial 필드 상세 비교 파일 |
